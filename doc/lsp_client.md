# Asynchronous LSP Client and IPC Architecture

To bring IDE features (like completions, hovers, jump-to-definition, and real-time compiler diagnostics) into a terminal environment, Alwide embeds a custom **Language Server Protocol (LSP)** client. Built in C, the client communicates with background servers (like `clangd` or `pyright`) asynchronously using low-level Unix Inter-Process Communication (IPC).

---

## 1. Process Lifecycles over Unix Pipes

The client handles language server lifetimes using Unix process controls and standard pipes.

```
  Alwide (Parent Process)                         LSP Server (Child Process)
   (e.g., inpipefd / outpipefd)                    (e.g., clangd, pyright)
        │                                                     │
        ├──► [Write Pipe] ──► content-length / JSON ──► [stdin]
        │                                                     │
        └──◄ [Read Pipe]  ◄── content-length / JSON ◄── [stdout]
```

### Forking and Pipe Initialization
During `LSP_openLSPServer()` inside [src/advanced/lsp/lsp_client.c](file:///home/arno/dev/Alwide/src/advanced/lsp/lsp_client.c):

1. **Creating Pipes**: Allocates two pipe descriptor pairs (`inpipefd` and `outpipefd`) using `pipe()` to direct bidirectional traffic.
2. **Process Forking**: Calls `fork()` to split execution:
   - **Child Process**: Calls `dup2()` to clone standard input (`stdin`) and standard output (`stdout`) to the pipe descriptors. It then calls `execvp()` to replace the child process image with the language server executable (e.g., launching `clangd --background-index`).
   - **Parent Process (Alwide)**: Closes the unused pipe endpoints, saving the active pipe read/write file descriptors into the `LSP_Server` context block.
3. **Graceful Terminations**: Upon shutdown, `LSP_closeLSPServer()` sends a `shutdown` JSON-RPC message, waits for exit confirmation, and kills residual processes cleanly using `kill(server->pid, SIGTERM)`.

## 2. JSON-RPC Communication Protocol

LSP uses JSON-RPC 2.0 messages. Alwide prefixes these payloads with a standard HTTP-like header specifying the size of the JSON body:
```http
Content-Length: 428\r\n
\r\n
{ ... JSON Payload ... }
```

### Packet Types
The client handles three distinct types of JSON-RPC packages:
1. **`LSP_REQUEST`**: Outbound queries (e.g., requesting autocomplete lists). Requests carry a unique `id` generated by incrementing `request_id`.
2. **`LSP_NOTIFICATION`**: One-way messages without a response tracking ID (e.g., notifying the server of cursor changes or text modifications via `textDocument/didChange`).
3. **`LSP_RESPONSE`**: Inbound answers returning compiler structures or errors.

### The Response Context Manager
To handle responses asynchronously without blocking user input, the client registers sent requests inside a context linked list:

```c
struct _LSP_ResponseContext {
  LSP_PacketID id;          // Target tracking ID
  char* method;             // Expected method (e.g., "textDocument/completion")
  char file_name[PATH_MAX]; // Absolute path of the active buffer
  void* payload;            // User-data context payload
  struct _LSP_ResponseContext* next;
};
```

When a response arrives from the background pipe, the editor pops the matching request context using `LSP_popResponseContext()`, parses the results, and dispatches them to the target UI.

### JSON Serialization via `cJSON`
The client delegates all document structures serialization to the lightweight `cJSON` library (`lib/cJSON/cJSON.c`). It serializes text coordinates to match LSP structures:
* **Coordinates Difference**: Alwide's internal buffer uses a **1-based absolute row / 0-based column** layout, whereas LSP requires a **0-based row / 0-based character** index. Conversion functions like `LSP_getJSONPosition()` convert these values seamlessly.

## 3. Supported Protocol Capabilities

The client processes five core protocol capabilities:

* **Document Synchronization**: Pushes document opening notifications (`textDocument/didOpen`) and registers incremental string adjustments (`textDocument/didChange`) whenever edits are recorded in `state_control`.
* **Code Completion (`textDocument/completion`)**: Triggers completion queries when typing. The server returns arrays of completions, which Alwide parses into `LSP_CompletionList` arrays to show in the dropdown window.
* **Hover Tooltips (`textDocument/hover`)**: Displays documentation tooltips. It parses markdown blocks returned from the server and prints formatted descriptions inside popup overlays.
* **Jump to Definition (`textDocument/definition`)**: Finds definition positions across files. If the definition is in another file, Alwide opens the document and positions the cursor at the target row/column.
* **Code Formatting (`textDocument/formatting`)**: Dispatches a formatting query before saving. The server returns formatting arrays, which Alwide translates into edit commands to format the buffer.

## 4. Asynchronous Multi-threading and Synchronization

Integrating background processes into single-threaded text loops requires thread-safe state protections:

* **POSIX Mutex Synchronization**: Since server subprocesses run independently, the editor protects pending queues using mutex locks (`pthread_mutex_t pending_lock`). 
* **Queuing Packets**: If the editor attempts to send a notification before server handshakes complete, Alwide adds the message to a thread-safe `pending_packets` linked list. Once initialized (`is_init_done`), a handler pushes the pending queue down the Unix write pipe.
* **Non-Blocking Pipe Scans**: To keep typing responsive, pipe checks must remain non-blocking. The main loop checks for data on the server stdout pipe using Unix `select()` or non-blocking standard reads (`LSP_readPacketAsJSON(server, false)`). If the pipe is empty, control returns to key listeners instantly.
* **Main Loop Paint Signals**: When an asynchronous response contains new diagnostics or completions, the client marks `ctx.refresh_local_vars = true` and `gui_doesGUINeedRepaint()`. The main loop intercepts these flags and schedules a repaint to display updates instantly.
